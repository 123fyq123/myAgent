# Lean API

## 目标

围绕一个具体功能，梳理“入口在哪里、路由怎么走、跨了哪些服务、落到了哪些数据结构和表”，并写成项目内可复用的 Markdown 文档。

这不是泛泛解释接口概念，而是要落到当前仓库的真实文件、真实调用链和真实验证方式。

## 输入约定

优先支持这两种调用：

- `/leanApi 功能名称`
- `/leanApi 模块名 功能名称`

解析规则：

- 如果用户显式给出模块名，例如 `user`、`project`，直接使用该模块。
- 如果用户只给功能名称，就先在仓库里检索并推断最可能的模块，然后在文档中写明推断依据。
- 模块名优先使用后端服务域名或业务域名，例如 `user`、`project`、`task`、`menu`、`department`。

## 必读上下文

开始前必须先读这些文件，不能跳过：

1. 根目录 `AI_INDEX.md`
2. 后端总览 `backend/ms_project/AI_CONTEXT.md`
3. 目标模块的 `AI_CONTEXT.md`

如果功能明显涉及前端页面或前端调用入口，再补充：

- `frontend/AI_CONTEXT.md`
- 若不存在，则查看 `frontend/src` 的 `api`、`router`、`views`、`store`

如果功能跨多个模块，至少把主模块和直接上下游模块的 `AI_CONTEXT.md` 读一遍。

## 执行流程

### 第一步：确定功能边界

先明确这次要回答的核心问题：

- 这是哪个业务功能
- 它属于哪个模块
- 用户真正想看的是“接口定义”、还是“全链路调用过程”、还是“两者都要”

然后提炼一组检索关键词：

- 中文功能名
- 可能的英文函数名、路由名、RPC 名
- 相关 DTO / VO / protobuf message 名

### 第二步：从入口往后追

优先按下面顺序找链路，能找到几层就写几层，不要硬编不存在的层：

1. 前端页面、按钮、弹窗或列表入口
2. `frontend/src/api` 中的 HTTP 请求封装
3. `project-api` 中的 Gin 路由、handler、参数绑定和鉴权
4. API 层发起的 gRPC client 调用
5. `project-user` 或 `project-project` 等服务内的 service 实现
6. repo / dao / model / 表
7. 相关 proto、pb、枚举、状态流转、缓存、消息或外部依赖

如果某个功能没有前端入口，例如纯后端接口、内部 RPC、定时任务，也要明确写出来。

### 第三步：只记录可证实的信息

必须基于实际文件、实际路径和实际调用关系写文档：

- 用 `rg` 搜索功能关键词、路由、方法名、proto 名
- 读取最小必要文件，不要无边界展开整个仓库
- 调用链必须能被源码中的 import、注册、调用或配置关系支撑

禁止：

- 凭经验脑补调用层
- 把“可能如此”写成“已经如此”
- 省略关键断点位置，只写抽象描述

如果某一段链路暂时找不到，要明确标注“当前仓库中未定位到”以及已检查的位置。

### 第四步：补齐全链路视角

文档至少尝试覆盖以下视角，按实际情况裁剪：

- 用户从哪里触发这个功能
- HTTP 接口是什么
- 是否经过网关鉴权、参数绑定、统一响应包装
- 是否调用 gRPC
- 下游服务里谁负责核心业务
- 是否涉及数据库读写、事务、缓存、消息
- 返回结果如何一路回到调用方

如果存在多个入口共享同一条后端链路，要在文档里合并说明，不要重复堆砌。

### 第五步：输出到 APIDocs

输出路径固定为：

- `APIDocs/<模块>/<功能目录>/README.md`

目录命名规则：

- 保留中文或常见技术缩写
- 去掉首尾空格
- 将空格、斜杠、反斜杠、冒号等路径不安全字符替换为 `-`

例如：

- `/leanApi user 登录` -> `APIDocs/user/登录/README.md`
- `/leanApi 任务看板` -> 若推断为 `project` 模块，可写入 `APIDocs/project/任务看板/README.md`

如果目标文档已存在：

- 先读取旧文档
- 增量更新已有章节
- 保留用户补充的有效内容
- 不要整篇覆盖成全新模板

## 文档模板

写文档前必须先读：

- [`references/api-chain-doc-template.md`](references/api-chain-doc-template.md)

最终文档要尽量覆盖模板中的核心章节，尤其不能删掉这些部分：

- 功能一句话说明
- 归属模块
- 链路总览
- 关键文件定位
- 详细调用链
- 调试与验证
- 注意点

## 质量要求

- 默认使用中文撰写
- 路径、命令、类型名、方法名保留原文
- 每个关键结论尽量对应到真实文件或目录
- 调用链要区分“已确认”与“待确认”
- 如果用了外部资料，必须附来源链接，并且只用于补充概念，不替代项目事实

## 推荐输出风格

- 先给“一眼看懂”的总结，再展开细节
- 多用表格列出“文件位置 -> 作用 -> 为什么相关”
- 调用链按步骤编号，便于后续排查
- 对容易踩坑的地方直接指出，不美化现状

## 交付说明

完成后，向用户简要说明：

- 文档写入路径
- 本次确认到的主链路范围
- 哪些部分已经确认，哪些部分还缺源码证据
