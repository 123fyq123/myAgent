# Project Learn

## 目标

以当前仓库作为学习主线，并在必要时通过 HTTP/网络查询官方或权威资料补齐背景，帮助用户学习一个没有接触过的知识点。不要做泛泛百科式回答；每次学习都要产出项目内 Markdown 知识文档，并尽量落到本项目的目录、文件、调用链、代码片段或运行方式上。

## 触发入口

当用户输入 `/learn <知识名>` 时执行本流程。若用户没有显式写 `/learn`，但请求是“学习/讲懂/带我理解某个本项目相关知识”，也按本流程执行。

## 边界

- 以当前项目为主线学习：优先使用 `AI_INDEX.md`、各模块 `AI_CONTEXT.md`、源码、配置、脚本和项目文档。
- 允许通过 HTTP/网络查询相关技术栈资料，尤其是 `pprof`、压测、Go runtime、Gin、gRPC、MySQL、Redis、Etcd、Nacos、Jaeger、Vue、Webpack 等项目无法单独讲清的知识。
- 外部资料只用于补齐背景、原理、命令参数、最佳实践或工具用法；最终解释必须回到本项目的落点、适用场景或可接入位置。
- 外部资料优先级：官方文档 > 标准库/项目仓库文档 > 成熟工具文档 > 权威技术文章。普通博客只能作为辅助，不要作为唯一依据。
- 使用外部资料时，要在答案中列出来源链接；如果没有联网能力或查询失败，要明确说明，并基于本项目和已有知识谨慎解释。
- 不修改业务代码，除非用户明确要求“边学边改/实现一个练习”。
- 不读取或复述敏感数据：遇到 `.env`、密钥、token、真实连接串、日志隐私内容时，只说明其用途和占位符含义。

## 文档产出

- 每次 `/learn <知识名>` 的主要产物必须是 Markdown 文档，而不是只在对话中解释。
- 输出路径固定为 `docs/learn/[知识名]/README.md`。例如 `/learn pprof` 写入 `docs/learn/pprof/README.md`，`/learn 压测` 写入 `docs/learn/压测/README.md`。
- 目录名从用户输入的知识名生成：去掉首尾空格，保留常见技术名大小写语义；空格、斜杠、反斜杠、冒号等路径不安全字符替换为 `-`。
- 如果目标目录或 `README.md` 不存在，创建它；如果已存在，先读取旧文档，再按本次新增理解增量更新，不要盲目覆盖用户已有内容。
- 文档模板见 `references/learn-doc-template.md`。产出时必须按该模板填充；可根据知识点裁剪不适用的小节，但不要删除“项目落点”“实践路线”“资料来源”这类核心部分。
- 如果知识点包含可执行操作，必须填充“通用操作步骤”：工具类写常用命令和参数说明，协议/框架类写标准开发或排查流程，避免只讲概念。
- 最终回复用户时，简要说明文档路径、是否查询了外部资料、是否存在未验证或待补充内容。

## 学习流程

1. 解析知识名，判断它更可能属于前端、后端、数据库/中间件、工程脚本或 Agent 资产。
2. 必须先读根目录 `AI_INDEX.md`。如果是后端知识，继续读 `backend/ms_project/AI_CONTEXT.md` 和相关模块的 `AI_CONTEXT.md`；如果是前端知识，读 `frontend/AI_CONTEXT.md`，不存在时查看 `frontend/src` 结构。
3. 用 `rg` 查找知识名、相关英文名、接口名、路由名、包名、表名或配置项。优先读最小必要文件，不要一口气展开无关模块。
4. 如果项目内资料不足以讲清概念、工具或实践，使用 HTTP/网络查询权威资料；查询时优先搜索“技术名 + 官方文档/official docs/标准库/项目名”。
5. 建立“从通用概念到项目落点”的路径：通用原理、项目中是否已使用、可接入位置、前端页面/接口、HTTP 路由、gRPC 调用、service、repo/dao、proto、数据库或中间件，按知识点相关性裁剪。
6. 读取 `references/learn-doc-template.md`，按模板组织知识文档。
7. 将文档写入 `docs/learn/[知识名]/README.md`；如已有旧文档，先合并再写入。
8. 最终回复用户时用中文简要说明产出位置，并尽量引用本地文件链接和行号；使用外部资料时补充来源链接。

## 文档结构

知识文档使用 `references/learn-doc-template.md`。核心内容必须覆盖以下信息，保持清晰但不要过度冗长：

- `一句话先懂`：用项目语境解释这个知识点解决什么问题。
- `通用背景`：当项目内资料不足时，用外部权威资料解释必要原理、命令或工具用法；不需要时省略。
- `通用操作步骤`：列出脱离本项目也适用的命令速查、操作流程或排查步骤；例如 `pprof` 写采集/分析命令，`gRPC` 写 proto 修改、生成代码、服务实现、客户端调用流程。
- `在本项目哪里出现`：列出 2-5 个关键文件或目录，说明各自角色。
- `调用链/数据流`：按步骤说明它在项目里如何运转。
- `核心概念`：解释用户需要先掌握的 3-6 个概念，避免堆术语。
- `怎么继续看`：给出下一步阅读路线或 1-2 个小练习。
- `注意点`：列出本项目里的坑、约定或容易误解处；没有就省略。
- `资料来源`：如果查询了外部资料，列出链接；没有使用外部资料就省略。

## 教学风格

- 先给直觉，再给细节；先讲“为什么”，再讲“怎么实现”。
- 面向没接触过该知识的学习者，避免默认用户已经懂框架、协议或架构术语。
- 使用类比可以，但类比后必须回到项目代码。
- 如果代码里有历史包袱、命名不一致或实现不完整，要温和指出，不替项目“美化现实”。
- 如果无法在项目内找到知识点，说明已查找的关键词和位置，并给出最接近的项目落点。

## 示例

用户：`/learn gRPC`

应先定位后端工作区和 proto/gRPC 模块，再解释本项目中 `project-api` 如何通过 gRPC 调用 `project-user` 或 `project-project`，并给出 proto、生成代码、service 实现和调用侧的阅读路径。

用户：`/learn Vuex`

应先定位前端上下文，再解释本项目中 Vuex 如何保存用户、路由、权限或页面状态，并连接到相关 `store`、`router`、`api` 和页面组件。

用户：`/learn pprof`

应先查询 Go 官方 `net/http/pprof` 或运行时分析资料，解释 CPU、内存、goroutine、阻塞分析的用途，再回到本项目的 Go 服务结构，说明它目前是否接入、适合接入哪个服务、调试环境如何暴露以及生产环境需要怎样限制访问。最终写入 `docs/learn/pprof/README.md`。

用户：`/learn 压测`

应先补充压测基本概念和常见工具（如 `wrk`、`ab`、JMeter、k6）的适用场景，再结合本项目的 `project-api`、gRPC 服务、MySQL/Redis/Etcd/Nacos 等依赖，说明应该压测哪些入口、关注哪些指标，以及如何避免把依赖环境打崩。最终写入 `docs/learn/压测/README.md`。
